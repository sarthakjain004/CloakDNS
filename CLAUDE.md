## Project Overview

CloakDNS is a custom DNS server that intercepts all DNS queries on a machine or network, blocking tracker/ad domains before a connection is ever made. It goes beyond Pi-hole by handling CNAME cloaking, DoH/DoT bypass detection, cookie syncing hub blocking, and DNS traffic analysis resistance.

**Language:** C++
**Build system:** CMake (modern, C++20)
**Testing:** GoogleTest + CTest
**Target platforms:** Windows, Linux

---

## Architecture

```
Internet Traffic
      |
[CloakDNS - UDP port 53]
      |
      +-- Blocklist Engine (domain matching, wildcard support)
      +-- CNAME Uncloaker (recursive CNAME chain resolution)
      +-- DoH/DoT Bypass Detector (port 443/853 monitoring)
      +-- Cookie Syncing Hub Blocker (priority tracker domains)
      +-- Query Padding Module (EDNS0 padding for privacy)
      +-- DNS Cache (TTL-aware, with timing normalization)
      +-- Query Logger (structured JSON logs for dashboard)
      |
[Upstream DNS Resolver] (Cloudflare 1.1.1.1 / Quad9 / custom)
```

## Feature Set

### Core Features (from Project Plan)

1. **Domain blocking** -- Resolve known tracker/ad domains to 0.0.0.0 using community blocklists (EasyList, StevenBlack hosts, OISD, Disconnect, EasyPrivacy)
2. **CNAME uncloaking** -- Recursively resolve CNAME chains and block if any hop matches a tracker domain. Safari is the only browser with partial support; Chromium has none. CNAME cloaking is actively used by Adobe, Criteo, and Oracle to disguise third-party tracking as first-party (Dao et al., IEEE TNSM 2021; Vekaria et al. SoK 2025)
3. **DoH/DoT bypass detection** -- Smart devices (Alexa, Google Home, smart TVs) use hardcoded DNS servers (8.8.8.8) or DNS-over-HTTPS to skip local DNS entirely. Detect outbound DNS on port 443/853 and alert/redirect (XDA 2026 report)
4. **Structured logging** -- Every blocked/allowed query logged with timestamp, domain, query type, action, and CNAME chain for the analytics dashboard (Sub-Project 6). JSON format for easy ingestion

### New Features (from Research Papers)

5. **Cookie syncing hub blocking (priority tier)** -- Research shows cookie syncing amplifies tracking exponentially. A single cookie from adverticum.net was synced to 82 parties covering 752 of the top 1,000 sites (Englehardt & Narayanan, CCS 2016). Blocking syncing hubs is far more effective than any cookie policy. Maintain a priority blocklist of syncing hubs:
   - doubleclick.net (syncs with 118 parties), google-analytics.com (609K sites), googlesyndication.com, googleadservices.com
   - facebook.net, fbcdn.net (widget tracking on 309K sites)
   - adnxs.com (AppNexus, 12% of top sites)
   - rubiconproject.com, openx.net, turn.com (synced ID with 43 domains)
   - scorecardresearch.com (ComScore), mathtag.com, advertising.com
   - adsrvr.org (30 domains), mookie1.com (29 domains), crwdcntrl.net, media6degrees.com
   - gemius.pl (33 IDs synced), 2o7.net/omtrdc.net (Adobe/Omniture)
   - rlcdn.com, adverticum.net

6. **Fingerprinting script domain blocking** -- FPTrace (WWW 2025) proved fingerprinting drives ad targeting. Block known fingerprinting providers:
   - doubleverify.com (7,806 sites), lijit.com (2,858 sites), alicdn.com (904 sites)
   - addthis.com/ct1.addthis.com (5,282 sites -- 95% of canvas FP), ligatus.com
   - audienceinsights.net, boo-box.com, cdn.augur.io
   - static.fraudmetrix.cn, cdn-net.com, js.ad-score.com, pxi.pub

7. **Evercookie infrastructure blocking** -- Evercookies use multiple storage mechanisms and regenerate from any survivor (Soltani et al.; Kamkar). Block known respawning domains:
   - bbcdn-bbnaut.ibillboard.com (respawned 69 cookies across 24 sites)
   - kiks.yandex.ru, irs01.net, simg.sinajs.cn
   - Known evercookie vendors: iovation.com, threatmetrix.com

8. **Data broker tracking domain blocking** -- Block data broker infrastructure that links online/offline identifiers:
   - LiveRamp/Acxiom AbiliTec domains (identity resolution)
   - Oracle Data Cloud / BlueKai domains (OIDG cross-device graph)
   - Criteo, TikTok Events API endpoints, Meta Conversions API endpoints

9. **CSS-triggered exfiltration blocking** -- CSS-only fingerprinting (NDSS 2025 "Cascading Spy Sheets") works without JavaScript by triggering conditional DNS lookups via @import, @media, background-image:url(). CSS container queries can fingerprint OS/CPU via calc() floating-point differences. 81% of email clients fetch remote CSS resources without user interaction. Blocking the tracker domain at DNS level prevents all CSS-based exfiltration with zero overhead (vs their proposed 30% overhead mitigation)

10. **Extension data exfiltration blocking** -- 3,028 Chrome extensions (2.68%) automatically exfiltrate user data, 68% via web requests to remote domains (Arcanum, 2024). 65 extensions use unencrypted HTTP. Block known data-harvesting endpoints used by malicious extensions (Sentry endpoints, ad-tech domains). VPN extensions like Touch VPN (8M users), Hola VPN (6M users) also exfiltrate

11. **EDNS0 query padding** -- Siby et al. (NDSS 2020) proved that encrypted DNS (DoH/DoT) is vulnerable to traffic analysis -- adversaries identify websites with 90%+ accuracy from packet sizes alone, using 124x less data than HTTPS fingerprinting. Standard EDNS0 padding to multiples of 128/468 bytes reduces accuracy but doesn't eliminate it. CloakDNS should pad outgoing queries to a fixed block size to reduce traffic analysis surface. Note: padding alone is insufficient (F1 still 0.43-0.69), but it's a useful layer

12. **DNS cache with timing normalization** -- The Performance API exposes domainLookupStart/domainLookupEnd timestamps that scripts abuse for DNS-timing fingerprinting (FP-Radar, PETS 2022). CloakDNS's local cache naturally reduces timing variation. Additionally, adding small random jitter to cached responses can further normalize timing signals

13. **Wildcard and regex domain matching** -- Support `*.tracker.com` patterns and regex rules for flexible blocking. Essential because trackers use randomized subdomains (e.g., `a1b2c3.tracker.com`)

14. **Allowlist/passthrough mode** -- Per-domain or per-application allowlisting so legitimate services sharing tracker infrastructure (e.g., googleapis.com, amazonaws.com, cloudfront.net) are not broken

15. **Hierarchical tracker blocking** -- Roesner et al. (NSDI 2012) showed advertising networks form hierarchies (site -> AdMeld -> Turn.com -> InviteMedia). Blocking the parent domain kills all child trackers. Maintain parent-child tracker relationship data

16. **Tracker behavior classification** -- Implement Roesner's five-behavior model for intelligent blocking:
    - Type A (Analytics): within-site only (GA) -- optional blocking, lower privacy threat
    - Type B (Vanilla): classic cross-site via third-party cookies -- block
    - Type C (Forced): forces first-party position via redirect -- DNS blocking is primary defense
    - Type D (Referred): piggybacks on another tracker -- blocking parent kills it
    - Type E (Personal): sites users visit that also track (Facebook widgets) -- block widget subdomains only

17. **GPC signal awareness** -- Only 44-45% of sites honor Global Privacy Control opt-out signals (GPC Compliance study, 2025). 90% of crawled sites integrate ad/tracking services. DNS-level enforcement is categorically more reliable than consent-based mechanisms. Log when blocked domains are from sites known to ignore GPC

18. **DNS-over-Tor forwarding (optional)** -- Siby et al. showed Tor is the most effective defense against DNS traffic analysis (F1 drops from 0.91 to 0.033). Offer optional mode to route DNS queries through Tor. Trades latency for maximum privacy. Cloudflare offers a .onion resolver

19. **Multi-protocol upstream support** -- Forward queries to upstream resolvers via:
    - Standard DNS (UDP/TCP port 53)
    - DNS-over-TLS (port 853)
    - DNS-over-HTTPS (port 443)
    User chooses based on their threat model

20. **Blocklist auto-updater** -- Python companion script (already planned) should also:
    - Merge and deduplicate from EasyList, EasyPrivacy, StevenBlack, OISD, Disconnect, Peter Lowe's list
    - Pull CNAME cloaking lists (Dao et al. maintains one)
    - Pull fingerprinting domain lists (from FP-Radar, OpenWPM research)
    - Auto-update on configurable schedule
    - Track blocklist version/timestamp for reproducibility

## Research Justifications

### Why DNS-level blocking matters more than browser extensions

1. **Browser extensions can't see server-side tracking** -- conversion APIs (Meta, Google, TikTok) send data server-to-server, invisible to extensions (Vekaria SoK 2025: "APIs and signals become undetectable by client-side mechanisms")
2. **Browser extensions can't see CNAME-cloaked requests** -- browser treats them as first-party (only Safari has partial defense via CNAME+IP heuristics)
3. **Browser extensions can't stop CSS-only fingerprinting** -- no JS involved, just DNS lookups from CSS @import/url() (Cascading Spy Sheets, NDSS 2025)
4. **Browser extensions can't intercept extension data exfiltration** -- 3,028 extensions exfiltrate data via web requests (Arcanum 2024)
5. **Filter lists have significant gaps** -- Disconnect blocks only 17.6% of canvas fingerprinting scripts (Englehardt 2016). EasyList+EasyPrivacy blocks 25.1% of scripts / 88.3% of affected sites
6. **Do Not Track is useless** -- enabling DNT reduced synced domains by only 2.9% (Acar 2014)
7. **Opt-out doesn't stop fingerprinting** -- AddThis continued canvas fingerprinting after opt-out (Acar 2014)
8. **GPC compliance is unreliable** -- ~55% of sites with privacy strings ignore opt-out signals (GPC 2025)
9. **The SoK paper itself calls for "beyond the browser" defenses** -- "Future research must therefore look beyond 'fix it in the browser' remedies and explore complementary approaches" (Vekaria 2025)

### Why DNS blocking over other network-level approaches

1. **Zero client modification** -- works for every device on the network (phones, tablets, smart TVs, IoT)
2. **Minimal performance impact** -- DNS queries are tiny; blocking is a hash lookup
3. **Blocks before connection** -- no bytes wasted on tracker connections
4. **Covers all applications** -- not just browsers; any app using DNS

### What DNS blocking cannot stop (known limitations)

1. **First-party tracking** -- scripts served from the same domain as the site (need proxy-level defense, Sub-Project 3)
2. **TCP/IP passive fingerprinting** -- p0f-style OS detection from packet headers (need VPN/proxy)
3. **Shared CDN domains** -- can't block amazonaws.com or cloudfront.net wholesale without breaking legitimate services (need allowlisting)
4. **Server-to-server data sharing** -- once data reaches a tracker's server, backend merges are opaque
5. **Behavioral biometrics** -- keystroke/mouse patterns collected via first-party JS (need Sub-Project 5)
6. **DNS resolution timing fingerprinting** -- Performance API exposes lookup timestamps (cache + jitter helps but doesn't eliminate)

## Key Detection Heuristics (for future dynamic blocking)

These heuristics come from the research papers and could enable CloakDNS to dynamically identify unknown trackers:

1. **ID cookie pattern** (Englehardt 2016): Cookie expires >90 days, value length 8-100 chars, stable across visits, <66% Ratcliff-Obershelp similarity between different machines
2. **Cookie syncing signature** (Acar 2014): User IDs appearing in request/referrer/location URLs across different third-party domains during redirect chains
3. **CNAME cloaking detection** (Safari/ITP): First-party subdomain has CNAME pointing to a domain whose IP resolves to a known tracker network
4. **Fingerprinting domain signal** (FP-Radar): Third-party domain serves scripts that access Canvas, WebGL, AudioContext, Battery, Navigator, Network Information APIs
5. **Tracker prominence metric** (Englehardt 2016): Third-party domain appears on >0.1% of top-1M sites and sets persistent cookies

## Technical Approach

- **UDP/TCP server** on port 53 using raw sockets or Boost.Asio
- **DNS packet parsing** via mnezerka/dnslib (C++ library)
- **Reference implementation** StefanPenchev05/DNS-Server-Cpp for basic structure
- **Blocklist engine**: Trie or hash-set for O(1) domain lookups, with wildcard support via suffix matching
- **CNAME uncloaking**: On every query, recursively resolve the CNAME chain via upstream, check each hop against blocklist
- **EDNS0 padding**: Pad outgoing queries to nearest multiple of 128 bytes per RFC 8467
- **Config format**: TOML or YAML for blocklists, upstream resolvers, logging, padding settings
- **Worker thread pool** for concurrent query handling (reference: cpp-pro concurrency patterns)
- **Blocklist update script**: Python, fetches/merges/deduplicates from multiple sources on configurable schedule
- **System service**: Run as Windows Service / Linux daemon (reference: abogeorge/ServiceAndDaemon)
- **Testing**: GoogleTest for unit tests, integration tests against known tracker domains, sanitizer-clean builds (ASan, UBSan, TSan)

## Open Research Problems (Future Work)

1. **Dynamic tracker detection** -- Move beyond static blocklists using ML-based classification of DNS query patterns (inspired by FP-Radar's predictive approach)
2. **Server-side tracking inference** -- Detect server-side tracking (Meta Conversions API, server-side GTM) by observing DNS patterns that correlate with known server-side endpoints
3. **Encrypted ClientHello (ECH) interaction** -- How does TLS ECH affect CNAME uncloaking? Research needed
4. **DNS query multiplexing** -- Batch multiple queries to reduce traffic analysis surface (open problem from Siby 2020)
5. **Cross-device tracking via shared IP** -- Can CloakDNS detect and log when multiple devices generate correlated tracker queries, indicating potential cross-device linking?
6. **Adaptive padding** -- Dynamic padding strategies that minimize overhead while maximizing traffic analysis resistance (Siby showed standard padding is insufficient)
